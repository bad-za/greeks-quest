## Контекст проекта (прочитай перед началом)

**Что это**: Telegram Mini App «Greeks Quest» — обучающая игра по крипто-опционам.
React 18 + Vite 5 + TypeScript, **ноль runtime-зависимостей кроме react/react-dom** —
это осознанная философия, сохраняй её (dev-зависимости добавлять можно).

**Карта кода**:

| Путь | Что там |
|---|---|
| `src/lib/bs.ts` | Блэк-Шоулз (r=0): цены, греки, P&L позиций из ног |
| `src/lib/rng.ts` | mulberry32 + seeded GBM-генератор ценовых путей |
| `src/lib/telegram.ts` | обёртка Telegram WebApp SDK, деградирует вне Telegram |
| `src/lib/ask.ts` | клиент AI-тьютора (ходит в Cloudflare Worker) |
| `src/state/store.tsx` | XP/баланс/прогресс, localStorage + Telegram CloudStorage |
| `src/content/act[1-4].tsx` | весь учебный контент (уровни = данные + JSX) |
| `src/content/levels.ts` | реестр уровней `LEVELS` и актов `ACTS` |
| `src/content/types.ts` | типы Level / Step / Mission / MissionChoice / Verdict |
| `src/components/Mission.tsx` | движок миссий: выбор → анимация пути → вердикт |
| `src/components/LevelPlayer.tsx` | проигрыватель шагов уровня, начисление XP |
| `src/components/LevelMap.tsx` | карта курса, HUD, ачивки |
| `src/components/AskTutor.tsx` | модалка чата с тьютором |
| `worker/src/index.ts` | Cloudflare Worker: AI-тьютор, валидация Telegram initData |
| `.github/workflows/deploy.yml` | сборка + деплой на GitHub Pages при пуше в main |

**Команды**: `npm run dev` (localhost:5173, доступ без ограничений),
`npm run build` (= `tsc && vite build`). Воркер: `cd worker && npm run dev`
(нужен `worker/.dev.vars` с `DEV_MODE=1` и `ANTHROPIC_API_KEY=...`, см. `.dev.vars.example`).

**Стиль кода**: без точек с запятой, одинарные кавычки, 2 пробела, trailing commas.
Комментарии и все тексты UI — **по-русски**. Комментарии пишутся только там, где код
сам не объясняет ограничение.

**Правила для исполнителя**:
1. После каждого этапа: `npm test` и `npm run build` зелёные → отдельный коммит
   (сообщение по-русски, в стиле истории: `git log --oneline`).
2. Этапы, помеченные ⚠️, содержат шаги, требующие внешнего доступа
   (Cloudflare, BotFather, деплой в прод) — код пиши полностью, но сами
   внешние действия выполняй только с подтверждения пользователя; если
   подтверждения нет, оформи инструкцию в README и двигайся дальше.
3. Не добавляй runtime-зависимостей фронтенду. Не коммить `dist/` и `bot_token.env`.
4. Деплой фронтенда автоматический (пуш в main). Деплой воркера ручной:
   `cd worker && npx wrangler deploy` — только с подтверждения.

**Шаги**:

  - На экспирации (`day === dte`) стоимость ноги равна `payoffAtExpiry`.

## Этап 2. Контент-линт: курс не может учить неправильному

**Цель**: каждый уровень и каждая миссия проверяются автоматически. Главное —
вердикты (`best`/`ok`/`bad`) сверяются с реально посчитанным P&L на пути миссии.

**Файлы**: `src/content/content.test.ts` (новый).

**Шаги**:
- [ ] Структурные проверки (жёсткие, `expect`):
  - id уровней уникальны; id миссий уникальны; `act` каждого уровня существует в `ACTS`;
  - у каждого уровня ≥ 1 шага; в каждом квизе ≥ 2 вариантов и
    `0 <= correct < options.length`; `explain` непустой;
  - у каждой миссии ≥ 2 выборов, есть хотя бы один `best` и хотя бы один не-`best`;
  - для миссий с фиксированным путём: `path.fixed.length === days + 1` —
    **это защита от реального бага**: `Mission.tsx` читает `path[m.days]`,
    и короткий массив даст `undefined` → NaN в P&L. Если тест найдёт расхождение
    в `act4.tsx` — чини контент (добавь/убери точку или поправь `days`), сверяясь
    со смыслом сценария.
- [ ] Численные проверки (через `mission-math.ts` из этапа 1):
  - все премии ног конечны и `> 0`; P&L каждого выбора на финальном дне конечен;
  - P&L каждого выбора на дне 0 ≈ 0 (допуск $1);
  - **сверка вердиктов**: у каждой миссии финальный P&L выбора `best` строго больше
    финального P&L каждого выбора `bad`. Если нарушено — тест падает с сообщением,
    показывающим id миссии и все P&L.
- [ ] Для нарушений «best vs bad», которые окажутся **педагогически осознанными**
  (например, урок «плохое решение случайно сработало»), заведи в тесте явный список
  `KNOWN_ANOMALIES: string[]` (id миссий) с комментарием-обоснованием у каждого id.
  Пустой список — идеал. Добавлять туда id можно только после ручного разбора миссии.
- [ ] Соотношение `best` vs `ok` по P&L НЕ проверяем жёстко (вердикт оценивает качество
  решения, а не только исход) — вместо этого выведи таблицу P&L всех миссий через
  `console.table` в конце теста, чтобы её было видно в логах CI.

**Критерии приёмки**: `npm test` зелёный; если были правки контента — пройди
затронутые миссии в dev и убедись, что тексты дебрифов всё ещё соответствуют исходам.

**Подводные камни**: контент-файлы импортируют React-компоненты — это нормально,
JSX в Node-окружении просто создаёт объекты, рендерить ничего не нужно. Если
какой-то импорт всё же потянет `window` — чини по образцу `telegram.ts` из этапа 0.

## Этап 6. Честное слияние сейвов (CloudStorage ↔ localStorage)

**Цель**: игра с двух устройств не теряет прогресс. Сейчас
(`store.tsx`) побеждает сейв с бóльшим XP целиком — union-поля другого сейва

**Файлы**: `src/state/merge.ts` (новый), `src/state/merge.test.ts`, `src/state/store.tsx`.

**Шаги**:
- [ ] `merge.ts`: `export function mergeSaves(local: SaveState, cloud: SaveState): SaveState`.
  Правила:
  - «база» — сейв с бóльшим XP (при равенстве — local); из базы берутся `xp`,
    `balance`;
  - `completedLevels`, `achievements` — объединение (порядок: сначала база,
    затем новые из второго);
  - `progress` — поэлементный `Math.max` по каждому ключу обоих сейвов;
  - `busts` — `Math.max` (сумма задвоила бы счётчик при каждом слиянии).
- [ ] `merge.test.ts` — минимум сценарии:
  1. облако впереди по XP, но у локального есть уникальная ачивка → база облако,
     ачивка сохранена;
  2. устройства прошли разные уровни → в `completedLevels` оба, в `progress` max;
  3. облако = INITIAL (после сброса), локальный с прогрессом → побеждает локальный
     целиком;
  4. полное равенство → результат эквивалентен входу.
- [ ] В `store.tsx` заменить `setState(local => (cloud.xp > local.xp ? cloud : local))`
  на `setState(local => mergeSaves(local, cloud))`.
- [ ] Задел на будущее: в `SaveState` ничего не менять, но в `load()` уже есть
  `{ ...INITIAL, ...JSON.parse(raw) }` — этого достаточно, миграции не городить.

**Критерии приёмки**: тесты слияния зелёные; в dev прогресс сохраняется и
восстанавливается как раньше (localStorage-путь не изменился).

## Этап 8. ⚠️ Воркер + клиент: стриминг ответа тьютора

**Цель**: ответ появляется по мере генерации, а не после «думаю…» на 5–10 секунд.

**Файлы**: `worker/src/index.ts`, `src/lib/ask.ts`, `src/components/AskTutor.tsx`.

**Дизайн** (важно для совместимости): клиент шлёт в теле запроса флаг `stream: true`.
Воркер со флагом отвечает потоком `text/plain; charset=utf-8` (просто текстовые
чанки, без SSE-обёртки — fetch-стриминг с CORS этого не требует); без флага —
прежний JSON `{answer}`. Так порядок деплоя воркера и фронта не важен.

**Шаги**:
- [ ] Воркер: при `body.stream === true` использовать стриминг SDK:
  ```ts
  const stream = client.messages.stream({ model, max_tokens: 1024, system: SYSTEM, messages })
  const encoder = new TextEncoder()
  const readable = new ReadableStream({
    async start(controller) {
      try {
        for await (const event of stream) {
          if (event.type === 'content_block_delta' && event.delta.type === 'text_delta')
            controller.enqueue(encoder.encode(event.delta.text))
        }
      } catch { /* обрыв генерации — отдаём что успели */ }
      controller.close()
    },
  })
  return new Response(readable, {
    headers: { 'Content-Type': 'text/plain; charset=utf-8', ...cors(origin) },
  })
  ```
  Ошибки ДО начала стрима (авторизация, 4xx от API) — прежним JSON-путём.
- [ ] `ask.ts`: `askTutor(question, context, history, onDelta?: (full: string) => void)`.
  Если `Content-Type` ответа содержит `application/json` — прежняя ветка (ошибки и
  совместимость). Иначе читать `res.body.getReader()` + `TextDecoder('utf-8')` с
  `{ stream: true }`, накапливать текст, дёргать `onDelta(accumulated)`. Пустой
  итоговый текст трактовать как ошибку («Тьютор не ответил, попробуй ещё раз»).
- [ ] `AskTutor.tsx`: при отправке сразу добавлять пустое assistant-сообщение и
  обновлять его содержимое в `onDelta` (заменять последний элемент `msgs`).
  Индикатор «думаю…» показывать только пока не пришёл первый чанк. Логика отката
  при ошибке (вернуть вопрос в поле ввода) — сохранить.
- [ ] ⚠️ Деплой воркера — с подтверждения. Деплоить воркер ПЕРЕД пушем фронта
  (новый воркер совместим со старым клиентом, обратное неверно).

**Критерии приёмки**: в dev (`wrangler dev` + `npm run dev`) ответ тьютора
печатается постепенно; принудительная ошибка (выключить wrangler) показывает
прежнее сообщение об ошибке и возвращает вопрос в инпут.

## Этап 11. Монте-Карло: «а в 1000 параллельных вселенных?»

**Цель**: главный педагогический апгрейд. Одиночный seeded-путь миссии учит
«стратегия хороша, потому что путь оказался удачным» — ровно той ошибке, от которой
курс должен отучать. После вердикта показываем распределение исходов каждого выбора.

**Файлы**: `src/lib/mission-math.ts`, `src/lib/mission-math.test.ts`,
`src/components/Histogram.tsx` (новый), `src/components/Mission.tsx`,
`src/styles.css` (стили таблицы/гистограммы по образцу существующих `.stats`/`.mission-*`).

**Шаги**:
- [ ] В `mission-math.ts` добавить:
  ```ts
  /** Финальные P&L выбора на n альтернативных путях (только для seeded-миссий) */
  export function mcOutcomes(m: Mission, c: MissionChoice, n: number): number[]
  ```
  Реализация: `if ('fixed' in m.path) throw` (вызывающий обязан проверить);
  для i = 1..n построить `gbmPath({ ...m.path, seed: m.path.seed + i, spot: m.spot, days: m.days })`
  — те же vol/drift/**shockDay/shockPct** (шок — часть предпосылки сценария,
  случайность — всё остальное; зафиксируй это комментарием) — и вернуть
  `pnlAt(m, c, priceLegs(m, c), path[m.days], m.days)` по каждому пути.
  Премии входа считаются один раз — вход у всех вселенных одинаковый.
- [ ] Там же хелпер `mcStats(outcomes)`: `{ probProfit, median, p5, p95 }`
  (сортировка + индексы, без интерполяции).
- [ ] Тесты: детерминизм (два вызова — идентичные массивы); длина `n`; для лонг-колла
  при сильно положительном drift `probProfit` строго больше, чем при сильно
  отрицательном.
- [ ] `Histogram.tsx`: чистый SVG в духе `XYChart.tsx` — принимает `values: number[]`,
  ~24 корзины, бары зелёные при `x >= 0`, красные при `x < 0`, вертикальная линия
  на нуле, подписи min/median/max. Никаких зависимостей.
- [ ] `Mission.tsx`, фаза `result`, только если `'seed' in m.path`: кнопка
  `🌌 А в 1000 параллельных вселенных?`. По клику (лениво, один раз — `useState` +
  вычисление в обработчике):
  - `mcOutcomes` для **каждого** выбора миссии, N = 1000;
  - таблица: строка на выбор — название, P(профит) в %, медиана, «5%…95%»
    (usd-формат из `format.ts`), строка сделанного выбора подсвечена;
  - под таблицей `Histogram` для сделанного выбора;
  - поясняющий текст: «Вердикт оценивает решение в момент входа, а не исход.
    Здесь — 1000 переигрываний того же сценария с другой рыночной случайностью:
    смотри на форму распределения, а не на один результат.»
  - XP/баланс не трогаем — это просмотр, не сделка.
- [ ] Производительность: ~1000 путей × ≤60 дней × ≤4 ноги × ≤4 выбора — оценочно
  сотни тысяч вызовов `bs()`. Замерь `performance.now()`; если суммарно > 150 мс
  на телефонном троттлинге (эмуляция в devtools), снизь N до 500 — качество
  картинки не пострадает.

**Критерии приёмки**: тесты зелёные; в dev на миссии `m-first-pump` кнопка
показывает таблицу и гистограмму; в финальных исторических миссиях
(fixed-пути, `act4.tsx`) кнопки нет; повторный клик не пересчитывает.

## Этап 12. Сайзинг позиции в миссиях

**Цель**: портфель $10,000 перестаёт быть декоративным — размер позиции выбирает
игрок, и урок риск-менеджмента (акт IV) становится практикой: ×2 на плохом пути
реально приводит к ликвидации (механика `busts` в `store.tsx` уже есть).

**Файлы**: `src/components/Mission.tsx`, `src/lib/mission-math.ts` (если удобнее
масштабировать там), `src/styles.css`.

**Шаги**:
- [ ] В фазе `brief` над списком выборов — сегмент-контрол (компонент `Seg` из
  `ui.tsx`): `Размер позиции: ½× / 1× / 2×`, по умолчанию 1×. Состояние `sizeMult`.
- [ ] Масштабирование: при выборе строить ноги как
  `{ ...leg, qty: leg.qty * sizeMult }` и `spotQty * sizeMult` — то есть умножается
  количество, а не цены. Стоимость на карточках выбора («Стоимость: …») тоже
  пересчитывается с учётом множителя.
- [ ] `onComplete(verdict, pnl)` получает уже отмасштабированный pnl — ничего в
  `LevelPlayer` менять не нужно. XP от размера НЕ зависит (вердикт — про решение).
- [ ] На экране результата, если `sizeMult !== 1`, показать бейдж «размер ×2» рядом
  с P&L. Монте-Карло из этапа 11 должен использовать тот же множитель.
- [ ] Подсказка при 2×: маленькая строка под контролом — «двойной размер = двойная
  премия и двойной риск; портфель помнит всё».

**Критерии приёмки**: тесты зелёные; в dev выбор ½×/2× меняет стоимость на
карточках ровно в 0.5/2 раза; проигрышная миссия на 2× снимает с баланса вдвое
больше; серия убыточных сделок доводит до ликвидации (баланс сбрасывается на
$10,000, `busts` растёт, на карте появляется строка «💀 Ликвидаций…»); replay
по-прежнему не трогает баланс.
